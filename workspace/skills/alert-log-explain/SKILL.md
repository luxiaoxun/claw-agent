# Alert or Attack Log Explain Skill

## Overview
本技能用于告警或攻击日志的解读，分析告警中的攻击结果、严重等级等信息是否正确，如果不正确给出正确结论和解释，并给出处置建议。

## When to Use
- 用户提供了一条日志（log）或告警（alert）内容，希望理解其含义
- 用户怀疑告警中的攻击结果（attack_res）、严重等级（severity）、攻击方向（attack_direction）等字段可能存在误判
- 用户需要判断某个安全事件是否为真实攻击（True Positive）或误报（False Positive）
- 用户希望获得针对特定告警的处置建议（如隔离、忽略、深入分析等）
- 用户需要解释某个字段的含义，或说明攻击链中的位置
- 用户对告警中出现的 IP、域名、进程、文件路径等指标感到疑惑，希望获得风险评估

注意事项
- 本技能不调用任何第三方工具，仅基于 AI 内置知识和对用户提供内容的理解进行分析
- 如果信息不足，应主动向用户询问缺失的关键字段（如 payload、完整告警 JSON）
- 对于不确定的判断，应明确标注“需要进一步确认”，避免误导
- 处置建议应考虑用户环境（如是否为生产环境、是否有 SOC 流程），保持可操作性

## How to Use

### Step 1: 解析告警/日志信息
从用户输入中提取关键字段，常见字段包括但不限于：

| 字段                             | 说明 | 示例                              |
|--------------------------------|------|---------------------------------|
| `attack_res`                   | 攻击结果（成功/失败/未知） | success, fail, unknown          |
| `severity`                     | 严重等级 | critical, high, medium, low     |
| `attack_direction`             | 攻击方向 | inbound, outbound, lateral      |
| `src_ip`/`dst_ip`              | 源/目标 IP | 192.168.1.1                     |
| `attack_type`                  | 攻击类型 | SQL Injection, RCE, Brute Force |
| `rule_name`                    | 告警规则名称 | 疑似永恒之蓝攻击                        |
| `payload`/`request`/`response` | 攻击载荷或请求内容 | GET /?id=1' OR '1'='1           |
| `event_time`                   | 发生时间 | 2026-01-01 10:00:00             |
| `hostname` / `asset`           | 受影响资产 | web-server-01                   |
如果用户提供的内容缺少必要字段，主动询问补充。

### Step 2: 分析攻击结果与严重等级
对告警中的关键判断进行复核，给出正确性结论：

#### 2.1 攻击结果（attack_res）分析

| 告警中的结果 | 分析要点 | 可能的正确结论 |
|--------|----------|----------------|
| 成功     | 检查 payload 是否真正执行成功，是否存在防御机制（WAF、输入过滤） | 可能为 success / false positive |
| 失败     | 检查是否有后续行为（如反弹 shell、文件写入），是否存在漏报可能 | 可能为 fail / success（漏报） |
| 企图/未知  | 结合上下文行为分析，查看同时间段的关联日志 | 需给出推断结论 |

判断原则：
- 有明确回显、文件落地、进程创建、外联行为 → 倾向于 success
- 仅有探测行为，无后续利用迹象 → 倾向于 fail 或 scan
- 明显为扫描器或自动化探测流量 → 可判定为无害或低危

#### 2.2 严重等级（severity）分析

| 告警中的等级 | 是否合理 | 调整建议 |
|-------------|----------|----------|
| critical | 仅当攻击成功且造成严重危害（RCE、数据泄露、权限提升）时合理 | 如仅为探测或失败攻击，建议降级为 medium/high |
| high | 攻击可能成功或存在高风险漏洞利用 | 如攻击失败且无敏感数据风险，可降级为 medium |
| medium | 中等风险行为，如扫描、弱口令尝试 | 如确认为真实成功攻击，建议升级为 high/critical |
| low | 信息泄露、异常流量探测 | 通常合理 |
